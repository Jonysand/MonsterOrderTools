## Context

当前项目是一个B站直播弹幕处理工具，使用Windows SAPI进行语音合成。需要将其替换为小米MiMo-V2-TTS API，以获得更好的语音质量和更丰富的功能。

### 当前架构
- TTS管理器：`TTSManager`类（单例模式）
- 实现文件：`TextToSpeech.h`和`TextToSpeech.cpp`
- 依赖：Windows SAPI（`sapi.h`）
- 配置：通过`GET_CONFIG`宏获取配置项
- 调用方式：`Speak()`方法播放语音

### 约束
- 必须保持现有的API接口不变，避免修改调用代码
- 需要支持异步API调用，避免阻塞主线程
- 需要处理网络错误和API限流
- 需要管理API Key的安全存储

## Goals / Non-Goals

**Goals:**
- 替换Windows SAPI为小米MiMo-V2-TTS API
- 支持自然语言风格控制（如"激昂慷慨"、"温柔轻声"）
- 支持多方言（东北话、粤语、四川话等）
- 支持角色扮演语音合成
- 保持现有TTSManager接口兼容性
- 提供配置界面管理API Key和语音风格

**Non-Goals:**
- 不支持歌声合成功能（可后续扩展）
- 不实现实时语音合成（接受一定延迟）
- 不支持离线模式（需要网络连接）
- 不修改现有的弹幕处理逻辑

## Decisions

### 1. API集成方式
**决策**: 使用现有的`Network::MakeHttpsRequest`函数调用小米MiMo-V2-TTS API
**理由**: 项目已有成熟的HTTP客户端实现，支持异步协程，无需引入新依赖
**替代方案**: 
- 使用libcurl：需要新增依赖，增加项目复杂度
- 使用WinHTTP直接调用：现有实现已封装，无需重复开发

### 2. 音频播放方案
**决策**: 使用Windows多媒体API（`mciSendString`）播放MP3音频
**理由**: Windows原生支持，无需额外依赖，简单可靠
**替代方案**:
- 使用DirectShow：更复杂，但功能更强大
- 使用第三方音频库：增加依赖，但提供更多控制
- 使用Windows.Media.Playback：UWP API，不适用于Win32应用

### 3. 异步处理策略
**决策**: 使用现有的协程框架（`NetworkCoroutine`）处理异步API调用
**理由**: 与项目现有架构一致，代码风格统一
**替代方案**:
- 使用std::async：需要管理线程池，增加复杂度
- 使用Windows线程池：更底层，但难以维护

### 4. 配置管理
**决策**: 扩展现有配置系统，新增小米MiMo相关配置项
**理由**: 保持配置管理的一致性，用户无需学习新配置方式
**替代方案**:
- 使用独立配置文件：增加配置管理复杂度
- 使用环境变量：不够灵活，不便于用户修改

### 5. API Key安全
**决策**: 将API Key存储在本地配置文件中，不进行加密
**理由**: 本地桌面应用，用户自行管理安全风险
**替代方案**:
- 使用Windows凭据管理器：更安全，但增加实现复杂度
- 使用加密存储：增加复杂度，对用户透明度低

### 6. 功能开关设计
**决策**: 提供编译时宏和运行时配置双重控制机制
**理由**: 
- 编译时宏适用于无网络环境或不需要小米MiMo的场景
- 运行时配置提供灵活的引擎切换能力
**实现方式**:
- **编译时宏** `USE_MIMO_TTS`: 在`framework.h`中定义，默认启用。设为0可完全排除小米MiMo相关代码
- **运行时配置** `TTS_ENGINE`: 支持三种值
  - `"mimo"`: 强制使用小米MiMo
  - `"sapi"`: 强制使用Windows SAPI
  - `"auto"`: 优先使用小米MiMo，不可用时自动降级到SAPI
- **自动降级触发条件**:
  - 网络连接失败
  - API认证失败
  - API请求连续失败3次
  - API返回限流错误

## Risks / Trade-offs

### 风险1：网络延迟影响用户体验
**风险**: 从本地TTS转为网络API调用，可能增加延迟（1-3秒）
**缓解**: 
- 使用异步处理，不阻塞主线程
- 实现请求队列，避免并发请求过多
- 添加加载提示，让用户知道正在合成语音

### 风险2：API服务不可用
**风险**: 小米MiMo API服务可能不稳定或限流
**缓解**:
- 实现重试机制（最多3次）
- 添加本地缓存，对相同文本复用音频
- 提供降级方案（如使用Windows SAPI作为备份）

### 风险3：成本控制
**风险**: API调用可能产生费用，超出预算
**缓解**:
- 实现调用统计和日志
- 添加配置选项限制每日调用次数
- 优先使用免费额度

### 风险4：音频格式兼容性
**风险**: 小米MiMo返回的音频格式可能与播放器不兼容
**缓解**:
- 使用MP3格式（广泛支持）
- 实现格式转换逻辑（如需要）
- 添加音频播放错误处理

## Migration Plan

### 阶段1：准备工作
1. 在小米MiMo平台注册账号，获取API Key
2. 测试API调用，验证连通性
3. 更新配置系统，新增小米MiMo配置项

### 阶段2：实现核心功能
1. 创建`MimoTTSClient`类，封装API调用逻辑
2. 实现音频下载和本地存储
3. 实现音频播放功能
4. 集成到现有`TTSManager`

### 阶段3：功能扩展
1. 实现语音风格配置
   - 使用 `<style>风格词</style>` 标签将风格描述嵌入到文本前面
   - 支持预设风格：温柔轻声、激昂慷慨、新闻播报、欢乐活泼、沉稳严肃
   - 默认风格不添加风格标签
2. 实现多方言支持
3. 实现角色扮演功能
4. 添加配置界面（MiMo 设置在 SAPI 前面）

### 阶段4：测试和优化
1. 单元测试API调用
2. 集成测试弹幕播报
3. 性能测试和优化
4. 用户验收测试

### 回滚策略
- 保留现有Windows SAPI代码，通过配置切换
- 如果小米MiMo API不可用，自动降级到Windows SAPI
- 提供配置选项让用户选择TTS引擎
