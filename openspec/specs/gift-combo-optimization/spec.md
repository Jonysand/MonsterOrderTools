# 礼物连击优化规格

## 概述

优化礼物连击播报逻辑，防止用户通过频繁赠送小礼物刷屏。

## 背景问题

当前 `HandleSpeekSendGift()` 依赖官方 `combo_info` 超时机制。用户故意1个1个送礼物时，每次都会触发播报，造成刷屏。

## 解决方案

### 机制1: 全局冷却时间

- 同用户同礼物在冷却时间内只播报一次
- 累加后续礼物数量到对应的动态连击条目
- 冷却时间到后，播报累计总数

### 机制2: 动态连击检测

- 同用户同礼物在窗口时间内多次赠送，即使不构成官方combo，也创建虚拟连击
- 首报：连击开始时播报一次
- 尾报：超时后播报累计总数

## 播报规则

| 场景 | 播报时机 | 示例 |
|------|----------|------|
| 小额单次（<3个） | 立即播报 | "感谢A赠送的1个鲜花" |
| 大额单次（≥3个） | 立即播报 | "感谢A赠送的5个鲜花" |
| 官方连击 | 首报+超时尾报 | 首报："感谢A开始赠送鲜花" → 尾报："感谢A赠送的10个鲜花" |
| 动态连击 | 首报+超时尾报 | 同官方连击 |

## 数据结构

```cpp
struct DynamicComboEntry {
    std::string combo_id;       // open_id + gift_id
    std::string uname;          // 用户名
    std::string gift_name;      // 礼物名
    int gift_num;               // 累计数量
    float combo_timeout;        // 剩余超时时间
    bool paid;                  // 是否付费
    bool firstReported;         // 是否已首报
    int64_t lastUpdateTime;    // 最后更新时间
};
```

## 常量（写死）

| 常量名 | 值 | 说明 |
|--------|---|------|
| GIFT_COOLDOWN_SECONDS | 5 | 礼物播报冷却时间（秒） |
| DYNAMIC_COMBO_WINDOW_SECONDS | 10 | 动态连击检测窗口（秒） |

## 实现要点

1. **冷却检查**：每次礼物到达时检查是否在冷却中
2. **首报时机**：连击开始时（非小额单次）播报首报
3. **尾报时机**：超时后播报累计总数
4. **内存管理**：定期清理过期条目和冷却映射
